# СУБД PostgreSQL для проекта "Лотерея"

В этом лонгриде освящен процесс развертывания и подготовки ИТ-инфраструктуры проекта "Лотерея" - СУБД PostgreSQL под управлением Patroni и хранилища состояний etcd.

## Актуальность

В условиях импортозамещения и необходимости укрепления цифрового суверенитета российским ИТ-компаниям сегодня недостаточно только разрабатывать высоконагруженные веб-приложения. Круг задач расширяется требованиями обеспечивать их высокую доступность даже при наступлении аварийных рисков, чтобы соблюдать заданный SLA для заказчиков и клиентов. Условия новой реальности, требования законодательства и растущие потребности пользователей создают острую необходимость акцентировать внимание разработчиков не только на качестве программного кода прикладного приложения, но и на цифровой ИТ-инфраструктуре этого продукта, поскольку любое прикладное приложение осуществляет CRUD операции над данными, которые должны быть доступны максимально возможное количество времени для пользователей данного прикладного решения.

К ключевым особенностям текущего прикладного решения "Лотерея" (проекта) можно отнести:
- необходимость обеспечивать высокую доступность сервиса даже в случае сбоя с целью продолжения генерации прибыли;
- уменьшить время простоя в случае аварии на одном или нескольких узлах кластера СУБД; 
- ускорить процесс восстановления на момент времени в прошлом (PITR);
- заложить в инфраструктуру потенциал для дальнейшего обслуживания, миграции на отечественные коммерческие форки PostgreSQL, а также обеспечения мониторинга и горизонтального масштабирования путем реализации одной из следующих стратегий развития - шардинг, подключение геоудаленных узлов СУБД с целью повышения катастрофоустойчивости сервиса;

Для решения поставленных задач и достижения минимально достаточного уровня high availability (HA) прикладного решения выбрана трехузловая конфигурация кластера PostgreSQL 18 в отказоустойчивом исполнении под управлением Patroni и хранилищем состояний кластера etcd, также состоящего из трех узлов с синхронной репликацией между нодами.

## Стек

### Аппаратное обеспечение

Виртуальные машины etcd (pc-r-etcd01n1, pc-r-etcd01n2, pc-r-etcd01n3):
- ОС: `RED OS 8.0.2 Standard`;
- CPU: `4`;
- RAM: `8 Gb`.

Виртуальные машины СУБД (mph-r-pg01n1, mph-r-pg01n2, mph-r-pg01n3):
- ОС: `RED OS 8.0.2 Standard`;
- CPU: `6`;
- RAM: `8 Gb`.

Отдельная виртуальная машина для DNS-сервера с выделенной доменной зоной.

### Программное обеспечение
- `PostgreSQL 18.3` - СУБД для хранения данных;
- `Patroni 4.1.0` - Менеджер High Availability кластера СУБД (автоматический failover);
- `etcd 3.6.6` - Хранилище состояний Patroni в формате ключ-значение;
- `vip-manager 4.0.0` - Автоматическое управление переездом виртуального ip адреса в случае переключения лидера.

## Краткий план работ

1. Создание DNS-записей;
2. Разворачивание и настройка виртуальных машин. Обновление пакетов;
3. Установка и настройка кластера etcd из rpm-пакетов;
4. Установка СУБД PostgreSQL. Базовая настройка кластера СУБД и резервного копирования;
5. Установка и настройка Patroni на сервера СУБД. Подключение Patroni к etcd;
6. Проверка отказоустойчивости кластера СУБД - выполнение switchover средствами Patroni на узлах кластера;
7. (бонус) Симуляция отказа одного из узлов кластера СУБД.

## Результаты работы:

### 0. Установлена и сконфигурирована СУБД PostgreSQL в отказоустойчивом высокодоступном исполнении под управлением Patroni.

Используются 3 узла для СУБД и 3 узла для etcd.

Архитектура кластера баз данных позволяет проводить как вертикальное, так и горизонтальное масштабирование кластера. Возможна бесшовная интеграция аналитического функционала с кластером баз данных на реплику n3 (снижение нагрузки на лидер).

### 5. Настроено еженочное резервное копирование кластера СУБД в `/backup/basebackup_archive`

Пример на первом сервере:

```shell
[postgres@mph-r-pg01n1 ~]$ pwd
/var/lib/pgsql
[postgres@mph-r-pg01n1 ~]$ cat scripts/basebackup.sh
#!/usr/bin/bash
# скрипт для выполнения резервной копии кластера СУБД

DB_HOST=localhost
DB_PORT=5432
DB_NAME=postgres
BACKUP_DIR=/backup/basebackup_archive
BACKUP_USER=backup

# в учебных целях для имитации забора резервной копии в хранилище резервных копий
# перед созданием резервной копии удаляем содержимое целевой папки,
# имитируя забор РК
rm -rf $BACKUP_DIR/*

echo "$(date) - запущен pg_basebackup"
/usr/pgsql-18/bin/pg_basebackup \
    --pgdata="$BACKUP_DIR" \
    --format=tar \
    --wal-method=stream \
    --dbname="host=$DB_HOST port=$DB_PORT user=$BACKUP_USER dbname=$DB_NAME" \
    --no-password \
    --verbose && \
    echo "$(date) - выполнен pg_basebackup" || echo "$(date) - выполнено неуспешно"

[postgres@mph-r-pg01n1 ~]$ crontab -l
# Полное резервное копирование скриптом ежедневно в 3 часа ночи
0 3 * * * /var/lib/pgsql/scripts/basebackup.sh >> /log/basebackup.log 2>&1
[postgres@mph-r-pg01n1 ~]$
```
